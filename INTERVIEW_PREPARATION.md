## 🏗️ Структура проекта

### Технологический стек
- **Backend Framework**: Spring Boot 3.4.0
- **Язык**: Java 17
- **Сборка**: Maven
- **База данных**: PostgreSQL (RDS на AWS)
- **Кэширование**: Caffeine Cache
- **ORM**: Spring Data JPA / Hibernate
- **Контейнеризация**: Docker (multi-stage build)
- **Инфраструктура**: AWS (EC2, RDS, CloudFormation)
- **Внешние API**:
  - Telegram Bot API
  - Google Calendar API (OAuth 2.0)
  - OpenAI API (ChatGPT)
  - AssemblyAI API (Speech-to-Text)

### Ключевые компоненты

#### 1. TelegramCalendar (Основной класс бота)
- Наследуется от `TelegramLongPollingBot`
- Обрабатывает входящие сообщения и callback queries
- Управляет состояниями сессий через кэш
- Реализует паттерн State Machine для диалогов

#### 2. GoogleCalendarService
- Создание событий в календаре
- Поиск событий по ключевым словам и периодам
- Аналитика (подсчёт событий и общей длительности)
- Управление OAuth 2.0 токенами (автоматическое обновление)

#### 3. ChatGPTHandler
- Адаптивная генерация JSON запросов (поддержка новых моделей GPT-5)
- Обработка ошибок с чтением errorStream
- Валидация API ключей
- Поддержка разных типов запросов (создание, поиск, аналитика)

#### 4. VoiceCommandHandler
- Конвертация голосовых сообщений в текст через AssemblyAI
- Поддержка множества языков (language detection)
- Speaker labels для распознавания нескольких говорящих

#### 5. CacheConfig
- **User Cache**: 100 записей, TTL 1 час
- **Session Cache**: 300 записей, TTL 1 час
- **Calendar Selection Cache**: 300 записей, TTL 1 час

#### 6. UserAuthData
- Управление OAuth токенами (access_token, refresh_token)
- Хранение пользовательских настроек (keywords, calendar)
- Оптимистичная блокировка через @Version

1. **Адаптивная интеграция с OpenAI API**
   - Поддержка новых моделей GPT-5 (max_completion_tokens вместо max_tokens)
   - Автоматическое определение типа модели и генерация соответствующего JSON
   - Обработка ошибок с чтением errorStream для детальных сообщений

2. **OAuth 2.0 управление токенами**
   - Автоматическое обновление access_token через refresh_token
   - Безопасное хранение токенов в PostgreSQL
   - Оптимистичная блокировка через @Version для предотвращения race conditions

3. **Производительность**
   - Кэширование пользовательских данных (Caffeine Cache)
   - TTL 1 час для всех кэшей
   - Ограничение размера кэшей для контроля памяти

4. **State Machine для диалогов**
   - Управление состояниями через кэш (REQUEST_AUTHORIZATION, REQUEST_SUBMIT, etc.)
   - Поддержка многошаговых диалогов
   - Очистка состояний после завершения операций

5. **Infrastructure as Code**
   - Полная автоматизация развёртывания через CloudFormation
   - Multi-stage Docker build для оптимизации размера образа
   - Автоматическая установка зависимостей и сборка через UserData скрипт

6. **Обработка ошибок**
   - Валидация API ключей перед запросами
   - Детальные сообщения об ошибках для отладки
   - Graceful degradation при недоступности внешних сервисов

7. **Безопасность**
   - credentials.json в .gitignore
   - Передача чувствительных данных через environment variables в Docker
   - Maintenance mode для ограничения доступа

## ❓ Типичные вопросы и ответы

### Вопросы о технологиях

**Q: Почему выбрали Spring Boot?**
> A: "Spring Boot обеспечивает быструю разработку и удобную интеграцию с внешними API через dependency injection. Также он предоставляет готовые решения для работы с базами данных через Spring Data JPA и конфигурацию через @ConfigurationProperties. Это позволило мне сосредоточиться на бизнес-логике, а не на инфраструктурном коде."

**Q: Почему PostgreSQL, а не MySQL или MongoDB?**
> A: "PostgreSQL был выбран из-за его надёжности и поддержки сложных типов данных. Для хранения OAuth токенов и пользовательских настроек реляционная модель подходит лучше, чем NoSQL. Также PostgreSQL хорошо интегрируется с Spring Data JPA и поддерживает оптимистичную блокировку через @Version, что важно для предотвращения race conditions при обновлении токенов."

**Q: Зачем использовали Caffeine Cache?**
> A: "Кэширование критично для производительности, так как бот часто обращается к базе данных для получения пользовательских данных и токенов. Caffeine Cache предоставляет гибкую настройку TTL и размера кэша, что позволяет контролировать использование памяти. Я настроил три отдельных кэша: для пользовательских данных, сессий и выбора календаря, каждый с оптимальными параметрами."

### Вопросы об архитектуре

**Q: Как организована архитектура проекта?**
> A: "Я использовал слоистую архитектуру с разделением ответственности:
> - **telegram** слой: обработка сообщений и callback queries
> - **calendar** слой: интеграция с Google Calendar API
> - **chatgpt** слой: обработка запросов к OpenAI
> - **data** слой: управление пользовательскими данными
> - **entity** слой: JPA сущности и репозитории
> 
> Также реализовал State Machine для управления диалогами через кэш, что позволяет обрабатывать многошаговые сценарии, такие как авторизация или настройка ключевых слов."

**Q: Как обрабатываются состояния диалогов?**
> A: "Я использую Caffeine Cache для хранения состояний сессий. Когда пользователь начинает определённый процесс (например, авторизацию), я сохраняю состояние в кэш с ключом `chatId + "_state"` и значением типа `REQUEST_AUTHORIZATION`. При следующем сообщении от пользователя я проверяю состояние и обрабатываю сообщение соответственно. После завершения операции состояние очищается из кэша."

**Q: Как обеспечивается безопасность?**
> A: "Я реализовал несколько уровней безопасности:
> 1. Чувствительные файлы (credentials.json) добавлены в .gitignore
> 2. API ключи передаются через environment variables в Docker, а не хардкодятся
> 3. OAuth токены хранятся в зашифрованной базе данных
> 4. Реализован maintenance mode для ограничения доступа во время обновлений
> 5. Использую HTTPS для всех внешних API запросов"

### Вопросы об интеграциях

**Q: Как работает интеграция с Google Calendar?**
> A: "Интеграция реализована через Google Calendar API v3 с использованием OAuth 2.0. Процесс включает:
> 1. Пользователь получает ссылку для авторизации через бота
> 2. После авторизации получаем authorization code
> 3. Обмениваем code на access_token и refresh_token
> 4. Токены сохраняются в PostgreSQL
> 5. При истечении access_token автоматически обновляем его через refresh_token
> 
> Это обеспечивает непрерывную работу без повторной авторизации."

**Q: Как обрабатываются голосовые сообщения?**
> A: "Процесс включает несколько этапов:
> 1. Получаем файл голосового сообщения через Telegram Bot API
> 2. Загружаем файл в AssemblyAI для конвертации в текст
> 3. AssemblyAI определяет язык автоматически и возвращает текст
> 4. Отправляем текст в ChatGPT с промптом для извлечения структурированных данных
> 5. ChatGPT возвращает дату, время, длительность и описание события
> 6. Показываем пользователю результат для подтверждения
> 7. После подтверждения создаём событие в Google Calendar"

**Q: Как ChatGPT понимает естественный язык?**
> A: "Я создал специализированные промпты для каждого типа запроса (создание события, поиск, аналитика). Промпты содержат:
> - Инструкции по формату ответа
> - Примеры правильных ответов
> - Контекст текущей даты и времени
> - Информацию о пользовательских ключевых словах
> 
> ChatGPT анализирует текст и возвращает структурированные данные в заданном формате, которые я затем парсю и использую для работы с календарём."

### Вопросы о деплое

**Q: Как развёрнут проект на AWS?**
> A: "Я использовал Infrastructure as Code подход через AWS CloudFormation:
> 1. Создал YAML шаблон, который определяет всю инфраструктуру
> 2. Шаблон создаёт EC2 инстанс (t3.micro), RDS PostgreSQL (db.t4g.micro), Security Groups, IAM роли
> 3. UserData скрипт автоматически устанавливает Docker, клонирует репозиторий, собирает образ и запускает контейнер
> 4. Приложение контейнеризировано через Docker с multi-stage build для минимизации размера
> 5. Все настройки передаются через environment variables
> 
> Это позволяет развернуть всю инфраструктуру одной командой и легко масштабировать при необходимости."

**Q: Почему использовали Docker?**
> A: "Docker обеспечивает:
> 1. Консистентность окружения между разработкой и продакшеном
> 2. Изоляцию зависимостей
> 3. Простое развёртывание и масштабирование
> 4. Multi-stage build позволяет создать минимальный образ (только JRE, без Maven)
> 
> Это особенно важно для Spring Boot приложений, где нужно только runtime окружение."

### Вопросы о проблемах и решениях

**Q: С какими проблемами столкнулись и как их решили?**
> A: "Основные проблемы:
> 1. **Изменения в OpenAI API**: Новые модели GPT-5 требуют max_completion_tokens вместо max_tokens. Решение: реализовал адаптивную логику определения модели и генерации соответствующего JSON.
> 2. **Истечение OAuth токенов**: Access tokens истекают через час. Решение: реализовал автоматическое обновление через refresh_token перед каждым запросом к Google Calendar API.
> 3. **Производительность**: Частые обращения к базе данных замедляли работу. Решение: внедрил Caffeine Cache с TTL 1 час для пользовательских данных.
> 4. **Обработка ошибок API**: Не всегда было понятно, почему запрос не прошёл. Решение: добавил чтение errorStream для получения детальных сообщений об ошибках."

**Q: Как обрабатываете ошибки внешних API?**
> A: "Я реализовал многоуровневую обработку ошибок:
> 1. Валидация API ключей перед запросами
> 2. Проверка HTTP статус кодов
> 3. Чтение errorStream для получения детальных сообщений от API
> 4. Логирование ошибок для отладки
> 5. Отправка понятных сообщений пользователю вместо технических деталей
> 6. Retry логика для временных сбоев (где применимо)"

### Вопросы о масштабировании

**Q: Как проект будет масштабироваться?**
> A: "Текущая архитектура позволяет масштабироваться несколькими способами:

> 2. **Вертикальное масштабирование**: Увеличение ресурсов EC2 инстанса

> 5. **Webhook вместо LongPolling**: Можно перейти на Webhook для лучшей масштабируемости

### Вопросы о бизнес-логике

**Q: Как работает система ключевых слов?**
> A: "Система ключевых слов позволяет пользователям персонализировать события:
> 1. **Обычные ключевые слова**: Добавляются в начало описания события (например, 'Mike' → 'Mike. Встреча')
> 2. **Составные ключевые слова**: Группы слов, которые считаются одним ключевым словом ('Partner1 Partner2' → одно ключевое слово)
> 3. **Ключевое слово по умолчанию**: Используется, если в запросе не указано ключевое слово
> 
> Это особенно полезно для общих календарей, где нужно различать события разных пользователей."

**Q: Как работает аналитика?**
> A: "Аналитика позволяет пользователям получать статистику по событиям:
> 1. Пользователь указывает период (даты начала и конца)
> 2. Опционально указывает ключевое слово для фильтрации
> 3. Бот запрашивает все события из Google Calendar за период
> 4. Фильтрует по ключевому слову (если указано)
> 5. Подсчитывает количество событий и общую длительность в часах
> 6. Возвращает результат пользователю
> 
> Это полезно для отслеживания времени, потраченного на определённые типы задач."
